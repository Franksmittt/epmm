Strategic Blueprint: Custom CRM and Client Portal Architecture for AS Brokers
The independent financial services and wealth management sector within the Republic of South Africa is currently navigating a profound and systemic digital transformation. This evolution is fundamentally propelled by shifting consumer behaviors, highly stringent regulatory frameworks enforced by the Financial Sector Conduct Authority (FSCA), and the relentless advent of sophisticated financial technology platforms.1 Within this intensely competitive macroeconomic landscape, AS Brokers CC operates as a well-established, highly specialized niche financial advisory and short-term insurance firm based in Alberton, Gauteng.1 Trading under the regulatory umbrella of FSCA license FSP 17273, the enterprise has systematically cultivated a differentiated market position over a period exceeding two decades, transitioning from a traditional brokerage into a sophisticated conduit for alternative private equity investments.1
The primary objective of this exhaustive architectural blueprint is to furnish a granular, strategically sophisticated roadmap for the development of a custom, realistic, and sustainable Customer Relationship Management (CRM) system and authenticated Client Portal. This development will utilize Cursor AI as an autonomous coding agent, building upon the firm's existing Next.js 15, Supabase, and Tailwind CSS technological stack.1 The proposed architecture must securely accommodate distinct authentication tiers for executive administrators, operational staff, and high-net-worth customers.1 Furthermore, it must centralize the management of leads, automated follow-ups, complex financial quoting, and omnichannel client communication while introducing exceptional, out-of-the-box features that are practical to deploy without incurring insurmountable technical debt.2
The macroeconomic environment in South Africa is characterized by persistent inflationary pressures, a prolonged cycle of elevated interest rates, and structural impediments to domestic growth, rendering traditional investment paradigms increasingly ineffective for wealth preservation.1 In response, AS Brokers leverages a highly coveted FSCA Category 1.8 (Securities and Instruments: Shares) license, allowing the firm to distribute unlisted alternative private equity products, specifically the Everest Wealth portfolio.1 The translation of these high-risk, illiquid private equity operations into fixed-income retail products requires a digital architecture that perfectly synthesizes institutional-grade security, mathematical determinism, and seamless user experience.1 This report dissects the required database schemas, human capital routing workflows, integration methodologies, and advanced generative artificial intelligence configurations necessary to build an unparalleled wealth management ecosystem.
Foundational Technological Architecture and Database Schema
The transition from a public-facing, fragmented lead generation website to a secure, authenticated enterprise CRM and Client Portal requires a rigorous approach to full-stack architecture. The existing technological foundation relies on Next.js 15 utilizing the App Router paradigm, TypeScript for strict type safety, and Tailwind CSS for utility-first styling.1 To support a sustainable CRM, the backend infrastructure must be deeply integrated with Supabase PostgreSQL, utilizing Drizzle ORM for schema management and the pgvector extension for advanced vector embeddings.1
The Next.js 15 React Server Component Paradigm
The engineering mandate for the AS Brokers CRM dictates an elite technical strategy that ruthlessly prioritizes performance, data security, and programmatic efficiency. The architecture must default entirely to the React Server Components (RSC) paradigm introduced in Next.js 15.1 By executing the rendering logic and database queries on the server infrastructure, the application delivers fully formed HTML documents directly to the client browser, significantly reducing the JavaScript bundle size and accelerating the First Contentful Paint (FCP).1
Crucially, the utilization of Server Components ensures that highly sensitive proprietary logic—such as the algorithms calculating the South African Revenue Service (SARS) progressive income tax brackets, the mathematical formulas governing the 20% Dividends Withholding Tax (DWT) arbitrage, and external API keys for CRM integrations—remains completely hidden from the public internet.1 Client components, explicitly denoted by the use client directive, must be surgically isolated to highly interactive leaf nodes within the user interface, such as Framer Motion layout transitions, dynamic Recharts data visualizations, and complex multi-step forms utilizing React Hook Form and Zod schema validation.1
Data Mutations via Server Actions
Historically, managing form submissions, lead generation, and CRM data synchronization required the creation of disparate REST API routes. The modern Next.js 15 architecture fundamentally shifts this approach by utilizing Server Actions.5 Server Actions allow asynchronous server-side functions to be invoked directly from client components or form actions seamlessly.5
When a financial advisor updates a client's risk profile or a customer submits a request for an Everest Wealth quotation, the client component passes the payload to a Server Action. This action first validates the payload against a strict Zod schema to prevent malformed data injections, and subsequently executes the database mutation via Drizzle ORM or dispatches the payload to an external integration like HubSpot.1 This architecture drastically simplifies the codebase, eliminates the overhead of manual fetch() wrappers, and guarantees that the execution context remains highly secure.1
Multi-Role Authentication and Row-Level Security
A financial CRM handling high-net-worth client portfolios and proprietary institutional data requires an uncompromising approach to multi-tenant isolation and access control. The architecture must support three distinct login tiers: the Owner/Admin tier, the Staff tier, and the Customer tier. Supabase Auth, combined with PostgreSQL Row-Level Security (RLS), provides the optimal framework for enforcing these boundaries at the database level rather than relying exclusively on application-level logic.7
Frictionless Client Authentication via Magic Links
The target demographic for AS Brokers includes retiring professionals and high-net-worth individuals who frequently experience password fatigue and cognitive overload when interacting with complex financial portals.1 To drive adoption and reduce onboarding friction, the Client Portal must implement passwordless authentication mechanisms, specifically Supabase Magic Links.9 When a client wishes to access their unified wealth dashboard, they provide their verified email address. The Supabase Auth service dispatches a cryptographically secure, time-bound link to their inbox, granting them immediate access upon clicking.10 This approach meets the user where they are and significantly enhances the perceived elegance of the digital experience.
Database Schema and Role-Based Access Control
The core of the CRM's security architecture relies on a centralized profiles table that extends the native Supabase auth.users system.11 A PostgreSQL database trigger must be engineered to automatically generate a corresponding profile record whenever a new user authenticates.11 This profiles table will house the critical role column, defining the user as an admin, staff, or client.
Row-Level Security (RLS) is a PostgreSQL feature that enforces access control by dynamically appending implicit WHERE clauses to every query based on the authenticated user's context.12 This guarantees that even if a frontend vulnerability occurs, the database engine will mathematically reject unauthorized data retrieval.13

To maintain the exceptional performance characteristics expected of a premium Next.js application, these RLS policies must be meticulously optimized. A common anti-pattern involves calling auth.uid() directly within the policy condition, which forces the database engine to evaluate the function for every single row scanned.15 The Cursor AI agent must be instructed to wrap all auth functions within a SELECT statement (e.g., (SELECT auth.uid())). This syntax compels the PostgreSQL optimizer to execute an InitPlan, effectively caching the user's ID for the duration of the query and reducing retrieval latency from hundreds of milliseconds to single digits, even when scanning massive datasets.12
Cryptographic Security via Supabase Vault
Financial CRMs frequently require the storage of highly sensitive data, including third-party integration tokens, private API keys, and sensitive client identification numbers. Storing this data in plaintext within standard tables presents an unacceptable security risk.16 The architecture must leverage the Supabase Vault, a specialized PostgreSQL extension that provides Transparent Column Encryption.16
The Vault utilizes Authenticated Encryption to encrypt secrets at rest on the disk, making them accessible only in decrypted form through a specific PostgreSQL view accessed via SQL functions.17 By utilizing the vault.create_secret() function, AS Brokers can securely store API keys for external services (such as HubSpot or Property24 scrapers) directly within the database.17 This ensures that sensitive credentials never leak into the frontend Next.js bundle and remain cryptographically secure even in the event of a catastrophic database dump.17

Lead Management and Bi-Directional HubSpot Synchronization
The current AS Brokers architecture relies on a basic "search-patch" mechanism via a HubSpot Private App to synchronize contact data.1 To establish a truly enterprise-grade CRM ecosystem, this must be elevated to a continuous, bi-directional synchronization model utilizing Next.js Server Actions and HubSpot Webhooks.3
Bidirectional synchronization ensures that data flows continuously between the custom Supabase database and the HubSpot marketing engine, maintaining perfect consistency regardless of where updates occur.3 If Johnny Farinha updates a client's status from "Prospect" to "Closed Won" within the bespoke Next.js CRM interface, a background Server Action immediately fires an API payload to HubSpot, updating the corresponding contact record.19 Conversely, if a prospect engages with an automated email sequence in HubSpot and books a meeting, a webhook triggers an endpoint in the Next.js application, which subsequently updates the local Supabase leads table and populates the advisor's internal dashboard.3
This seamless interoperability eliminates manual data entry, prevents data drift, and provides a unified 360-degree view of the customer relationship across both sales and marketing channels.3
Programmatic Lead Scoring: Fit versus Engagement
Processing hundreds of inbound inquiries generated by the interactive Calculator Hub requires an automated qualification system to prevent the advisory team from expending resources on misaligned prospects.1 The CRM must implement a sophisticated dual-axis lead scoring matrix that evaluates both demographic "Fit" and behavioral "Engagement".20
The "Fit" score evaluates explicitly stated demographic and financial qualifications captured via progressive profiling.1 If a user utilizes the Estate Duty Calculator and inputs a gross estate value exceeding R10 million, the Server Action dynamically updates a custom property in HubSpot, injecting significant points into the contact's Fit score to flag them as a high-net-worth individual.1
The "Engagement" score tracks continuous behavioral signals across the digital platform. Downloading the "EverFree Income From Capital" brochure, completing the multi-step Financial Health Quiz, or engaging with the Digital Wealth Assistant chatbot all trigger incremental engagement points.1 Once a user's combined score crosses a predefined threshold (e.g., 85 points), the CRM triggers an automated webhook, routing the highly qualified prospect directly to Albert Schuurman for immediate, high-touch outreach, while lower-scoring leads remain in automated educational nurture sequences.1
The Automated Quoting Engine and Product Mechanics
A primary operational bottleneck for financial advisors is the manual generation of complex, highly regulated investment quotations and policy schedules. The AS Brokers CRM must feature a proprietary automated quoting engine that translates the specific mathematical and legal mechanics of the Everest Wealth private equity portfolio into deterministic, client-facing proposals.1
Modeling Alternative Private Equity Products
The Everest Wealth product suite operates on rigid structural parameters that must be hardcoded into the CRM's quoting logic.1 The system must enforce absolute minimum investment thresholds of R100,000 across all voluntary capital portfolios utilizing Zod schema validation.1
When generating a quote for the 12.8% Strategic Income Portfolio, the CRM must computationally project a flat monthly dividend yield of 12.80% generated through Class A preference shares.1 Concurrently, the engine must isolate and calculate the 10% "loyalty bonus." The system logic must strictly mandate that this bonus is only modeled for payout upon the successful maturation of the 60-month contractual term, reflecting the holding company's strategy to retain operational liquidity.1
Conversely, when an advisor selects the 14.2% Onyx Income+ Portfolio, the CRM must automatically disable any UI elements related to the loyalty bonus, executing the direct mathematical trade-off that abandons the end-of-term bonus in favor of maximizing immediate monthly cash flow via Class J preference shares.1
Furthermore, the CRM must enforce the complexities of South African tax arbitrage. Because these alternative investments utilize unlisted preference shares, the returns are legally classified as dividends.1 The quoting engine must automatically apply a flat 20% Dividends Withholding Tax (DWT) to the gross yield, explicitly visually contrasting this resulting net cash flow against the punitive marginal income tax rates (up to 45%) applied to traditional interest-bearing bank deposits.1 This mathematical visualization provides the advisor with an undeniable conversion argument.
Structuring Compulsory Capital: The Amethyst Living Annuity
For clients deploying compulsory retirement capital, the CRM must model the Amethyst Living Annuity, underwritten by 27four Life Limited.1 Because this product is strictly governed by Regulation 28 of the Pension Funds Act, the quoting engine must implement specialized boundaries.1
The system must allow the advisor to model a targeted net return (historically referenced around 10.2% to 12.12%) while strictly bounding the client's income drawdown rate between the statutory minimum of 2.5% and the maximum ceiling of 17.5%.1 The engine must also factor in the 9% "Inflation Protector" capital bonus applied after the first five years, conditionally validating that 90% of the initial capital is allocated to the Everest Strategic Income Portfolio to trigger the benefit.1 Finally, the tax logic must pivot entirely; internal growth remains tax-free within the wrapper, and the CRM must apply progressive SARS personal income tax tables only to the physical cash drawn down as income.1
Asynchronous Document Generation via Trigger.dev and Puppeteer
Generating FAIS-compliant quoting documents containing complex Recharts visualizations and mandatory liquidity warnings requires significant computational overhead that can cause serverless functions to time out.1 The architecture elegantly resolves this by offloading the document generation to background jobs utilizing Trigger.dev v4.1
When an advisor finalizes a quote in the CRM, the system dispatches a payload to a Trigger.dev worker.1 The worker utilizes Puppeteer to launch a headless browser and navigate to an internal, non-indexed Next.js print route (e.g., /internal/pdf-report).1 This route renders the complex financial projections and amortization schedules as a highly branded, pixel-perfect HTML document.1 Puppeteer captures this DOM as a PDF document, ensuring all Recharts animations are disabled for static rendering.1 Finally, the worker utilizes the Resend API to automatically dispatch the finalized quotation directly to the client's email, attaching all necessary regulatory disclosures regarding the 120-day notice periods and 15% early exit penalties inherent in private equity.1

Add-On 1: The Semigration Liquidity Tracker
AS Brokers currently executes a highly successful content marketing strategy capitalizing on the "semigration" trend—the macroeconomic phenomenon where affluent individuals continuously relocate from the inland economic hub of Gauteng to the coastal municipalities of the Western Cape upon retirement.1 This geographic transition necessitates the liquidation of massive primary residences, creating highly liquid clients holding millions in surplus capital that requires immediate deployment into high-yield assets like the Everest Wealth portfolios.1
The CRM must implement an automated Semigration Liquidity Tracker. By integrating a South African property data API—such as the AfriGIS Property Search API or utilizing a targeted Apify web scraper connected to Property24 28—the system can programmatically cross-reference the registered residential addresses of existing Gauteng-based clients against newly listed properties on the market.
If a client's address appears as a new listing, the CRM instantly generates a high-priority alert on Johnny Farinha's dashboard. This advanced intelligence allows the advisor to proactively initiate contact during the exact window of financial transition, perfectly positioning the firm's living annuities or 14.5% Strategic Growth products to absorb the impending liquidity event before competing institutions are even aware of the client's intent to relocate.
Add-On 2: Persistent Generative UI Memory and Context
The existing digital architecture features a highly advanced "Digital Wealth Assistant," an AI chatbot powered by the Vercel AI SDK 3.0 capable of streaming interactive Generative UI components directly to the user.1 However, standard AI implementations suffer from ephemeral memory; when the user refreshes the page, the context is destroyed. High-net-worth clients expect their advisors—and by extension, their digital assistants—to possess perfect recall.30
The portal must implement persistent AI and UI state management.31 Utilizing the onSetAIState callback provided by the Vercel AI SDK, the application will capture the entire serializable JSON history of every client conversation.31 This includes the natural language queries, the specific deterministic tool calls executed (e.g., calculateEstateDuty), and the raw mathematical arrays returned by the server.31 This state is securely committed to the Supabase PostgreSQL database and linked via a foreign key to the client's authenticated profile.32
When the client returns to the portal days or weeks later, a Next.js Server Component retrieves this localized history from Supabase and hydrates the chat interface using the initialAIState property.31 The Digital Wealth Assistant seamlessly resumes the conversation, "remembering" the exact parameters of the client's previous tax simulations and annuity drawdown inquiries, delivering an unparalleled, continuous advisory experience.30

This engine aggregates various data points from the CRM to calculate a dynamic score out of 100. Sustainable Development Architecture via Cursor AI
The execution of this highly sophisticated, mathematically rigorous financial ecosystem requires a disciplined approach to software engineering. The developer must utilize Cursor AI not merely as an autocomplete tool, but as an autonomous architectural orchestrator, meticulously constrained by deterministic rules to prevent LLM hallucinations and technical debt.1
The .cursorrules Framework and Context Hierarchies
To ensure the generated codebase strictly adheres to the project's architectural mandates, a robust .cursorrules framework utilizing the .cursor/rules/*.mdc format must be established.1 These rules will act as the immutable laws governing the AI's behavior.1
* Rendering Mandates: The rules must explicitly force the AI to default to React Server Components (RSC) for all data fetching and UI rendering, restricting the use client directive strictly to isolated, interactive leaf nodes to protect proprietary logic and optimize performance.1
* Mutation Standards: The framework must mandate the exclusive use of Next.js Server Actions for all database mutations and HubSpot API integrations, strictly prohibiting the generation of legacy REST API routes.1
* Design Systems: The AI must be constrained to utilize Tailwind CSS v4 utility classes exclusively, specifically enforcing the established design tokens (e.g., bg-void, shadow-cta-glow-blue) to maintain the high-end aesthetic.1
Furthermore, to prevent context degradation when generating complex modules, the developer must employ a hierarchical "Memory Bank" strategy.1 Root-level markdown documents must define the overarching business logic—such as the explicit requirement that all Everest voluntary products possess a strict R100,000 minimum threshold.1 When utilizing Cursor's "Composer" mode for multi-file generation, the developer must explicitly reference these localized context files, securely anchoring the LLM's generational capabilities to the deterministic mathematical truths required by the FSCA.1
Test-Driven Agentic Deployment
Within the highly regulated domain of financial services, the tolerance for arithmetic or logic errors is absolute zero.1 A minor hallucination in a compounding simulation could result in massive nominal inaccuracies over a twenty-year retirement projection, violating the fiduciary standards of the FSP.1
Therefore, Cursor AI must be directed utilizing a strict Test-Driven Development (TDD) loop.1 Before the agent is permitted to write the implementation code for a critical component, such as the AmethystLivingAnnuityCalculator, it must first be prompted to generate a comprehensive suite of Jest unit tests.1 These tests must contain hardcoded, verified input-output pairs—for instance, asserting that an input of R1,000,000 yielding 12.8% strictly returns R102,400 in net cash flow after the flat 20% DWT deduction.1 Only after the test suite is established and committed to version control should the Composer agent autonomously iterate on the TypeScript implementation logic until all assertions pass flawlessly.1 This rigorous methodology guarantees the integrity, safety, and sustainability of the entire CRM and Client Portal architecture.
